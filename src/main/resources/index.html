设计模式学习指南

结合 AI 代码生成项目，深入理解核心设计模式

开篇：为何要学习设计模式？

设计模式是软件开发人员在特定上下文中解决常见问题的“最佳实践”和“可复用解决方案”。它们不是现成的代码，而是指导你如何组织代码以实现最佳结构、可维护性和灵活性的蓝图。

在你提供的 aiautocode 项目中，就巧妙地运用了多种设计模式来构建一个清晰、可扩展的系统。本指南将带你逐一解析这些模式，并展示它们是如何协同工作的。

门面模式 (Facade)

简化复杂，提供统一入口。

执行器模式 (Executor)

解耦调用与执行。

策略模式 (Strategy)

封装算法，自由切换。

模板方法模式 (Template)

定义骨架，定制细节。

1. 门面模式 (Facade Pattern)

核心思想：为系统中的一组复杂接口提供一个统一、简化的入口。它隐藏了系统的复杂性，并向客户端提供了一个可以访问系统的接口。

代码中的体现：AiCodeGeneratorFacade.java

在你的项目中，AiCodeGeneratorFacade 就是一个典型的门面。客户端（比如一个Controller或Service）只需要与这个类交互，而不需要关心背后复杂的代码生成、解析和保存过程。


// 客户端只需要调用这一个方法，传入用户需求和生成类型
public File generateAndSaveCode(String userMessage, CodeGenTypeEnum codeGenType) {
    // ... 内部逻辑 ...
    return switch (codeGenType) {
        case HTML -> {
            // 调用AI生成服务
            HtmlCodeResult htmlCodeResult = aiCodeGeneratorService.generateHtmlCode(userMessage);
            // 调用执行器保存
            yield CodeFileSaverExecutor.executeSaver(htmlCodeResult, CodeGenTypeEnum.HTML);
        }
        // ... 其他 case ...
    };
}

客户端无需知道 AiCodeGeneratorService、CodeParserExecutor 或 CodeFileSaverExecutor 的存在。所有复杂的协调工作都由 AiCodeGeneratorFacade 完成。

使用场景

  • 当你需要为一个复杂的子系统提供一个简单的接口时。
  • 当客户端与子系统之间存在大量依赖关系时，使用门面可以解耦。
  • 当你需要构建一个分层系统时，门面可以作为每层的入口点。

2. 执行器模式 (Executor Pattern)

核心思想：将任务的提交与任务的执行解耦。在你的代码中，它更像是一种调度器或分发器，根据类型选择合适的处理器来完成工作。这与命令模式有相似之处，即将一个请求封装成一个对象。

代码中的体现：CodeParserExecutor.java & CodeFileSaverExecutor.java

CodeParserExecutor 和 CodeFileSaverExecutor 负责根据 CodeGenTypeEnum（代码生成类型）来决定调用哪个具体的解析器或保存器。


// CodeParserExecutor.java
public static Object executeParser(String codeContent, CodeGenTypeEnum codeGenType) {
    return switch (codeGenType) {
        case HTML -> htmlCodeParser.parseCode(codeContent); // 选择HTML解析策略
        case MULTI_FILE -> multiFileCodeParser.parseCode(codeContent); // 选择多文件解析策略
        default -> throw new BusinessException(...);
    };
}

这个执行器将“执行解析”这个动作与具体的解析逻辑（是解析HTML还是多文件）分离开来。调用方（AiCodeGeneratorFacade）不需要关心具体的解析实现，只需命令执行器去“执行”即可。

使用场景

  • 当一个操作有多种不同的实现方式，需要根据条件动态选择时。
  • 希望将请求的发起者和执行者解耦，方便地增加新的执行逻辑。
  • 用于集中管理和调度一系列相关的操作。

3. 策略模式 (Strategy Pattern)

核心思想：定义一系列算法，将每一个算法封装起来，并使它们可以相互替换。策略模式让算法的变化独立于使用算法的客户。

代码中的体现：CodeParseStrategy.java 及其实现

这是策略模式最经典的体现。首先，定义一个策略接口：


// CodeParseStrategy.java - 策略接口
public interface CodeParseStrategy {
    T parseCode(String codeContent);
}

然后，为不同的解析场景提供具体的策略实现：


// HtmlCodeParser.java - 具体策略A
public class HtmlCodeParser implements CodeParseStrategy {
    @Override
    public HtmlCodeResult parseCode(String codeContent) { /* ... HTML解析逻辑 ... */ }
}

// MultiFileCodeParser.java - 具体策略B
public class MultiFileCodeParser implements CodeParseStrategy {
    @Override
    public MultiFileCodeResult parseCode(String codeContent) { /* ... 多文件解析逻辑 ... */ }
}

最后，由上下文（Context），也就是我们的 CodeParserExecutor，来决定在运行时使用哪个具体策略。

使用场景

  • 当你有许多相关的类，它们之间只有行为（算法）上的差异时。
  • 需要一个算法的多种变体，并且希望在运行时动态选择。
  • 为了避免使用大量的条件语句（if-else 或 switch-case）来选择算法。

4. 模板方法模式 (Template Method Pattern)

核心思想：在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中实现。模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。

代码中的体现：CodeFileSaverTemplate.java 及其子类

这个抽象类定义了保存代码的完整流程（模板方法 saveCode），但将一些具体的、可变的步骤交给了子类实现。


// CodeFileSaverTemplate.java - 抽象模板
public abstract class CodeFileSaverTemplate {

    // 这是模板方法，定义了算法的骨架，final确保子类不能修改流程
    public final File saveCode(T result) {
        // 1. 固定的步骤：验证输入
        validateInput(result);
        // 2. 固定的步骤：创建唯一目录
        String baseDirPath = buildUniqueDir();
        // 3. 变化的步骤：保存具体文件（由子类实现）
        saveFiles(result, baseDirPath);
        // 4. 固定的步骤：返回目录
        return new File(baseDirPath);
    }

    // 抽象方法，由子类实现具体细节
    protected abstract void saveFiles(T result, String baseDirPath);

    // 钩子方法或具体方法，子类可以覆盖
    protected void validateInput(T result) { /* ... 默认验证 ... */ }

    // ... 其他方法 ...
}

具体的子类，如 HtmlCodeFileSaver 和 MultiFileCodeFileSaver，只需要关注它们自己特有的 saveFiles 和 validateInput 逻辑，而无需重复编写创建目录等通用代码。

使用场景

  • 当多个类有共同的算法，但具体步骤有细微差别时。
  • 为了防止子类修改算法的核心结构，同时又允许它们进行定制。
  • 用于代码复用，将公共行为提取到父类中。

模式的协同工作：一窥全貌

这些设计模式在你的项目中不是孤立存在的，而是相互协作，共同构建了一个强大而灵活的系统。它们的协作流程如下：

请求处理流程：

  1. 门面 (Facade) 接收请求
    客户端调用 AiCodeGeneratorFacade.generateAndSaveCodeStream()，这是唯一的入口。
  2. 门面 (Facade) 委托执行器 (Executor)
    在流式处理的 doOnComplete 回调中，门面调用 CodeParserExecutor.executeParser() 来解析完整的代码字符串。
  3. 执行器 (Executor) 使用策略 (Strategy)
    CodeParserExecutor 内部根据 codeGenType，从 HtmlCodeParser 和 MultiFileCodeParser 这两个策略中选择一个来执行解析任务。
  4. 门面 (Facade) 再次委托执行器 (Executor)
    门面拿到解析结果后，接着调用 CodeFileSaverExecutor.executeSaver() 来保存文件。
  5. 执行器 (Executor) 使用模板方法 (Template Method)
    CodeFileSaverExecutor 根据类型，选择 HtmlCodeFileSaver 或 MultiFileCodeFileSaver。这两个类都是 CodeFileSaverTemplate 的子类，它们会执行定义好的模板方法 saveCode() 来完成保存。

联合使用的优势

  • 高内聚，低耦合：每个模式都解决了特定的问题，使得类的职责单一、清晰。它们之间的依赖关系明确，易于维护。
  • 高灵活性和可扩展性：
    • 想增加一种新的代码类型（比如Vue单文件组件）？只需：
      1. 添加一个新的 CodeParseStrategy 实现。
      2. 添加一个新的 CodeFileSaverTemplate 子类。
      3. 在两个执行器（Executor）的 switch 语句中增加一个case即可。
    • 完全不需要修改门面类或现有的策略和模板类。这就是“对扩展开放，对修改关闭”的开闭原则的完美体现。
